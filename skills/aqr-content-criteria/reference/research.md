# Research doc content criteria

Criteria for research docs — notes that inform decisions but are not specs. These are often drafted by an AI and pasted in, so the bar is flexibility, not a fixed template. Where the doc lives is a layout choice; this skill covers content only.

## 1. Basic requirements

- State the question the note is answering, up front.
- Keep references — every claim that is not common knowledge links its source.
- Divide the doc by topic, one section per topic, so a reader can jump to what they need.
- Keep a summary or list of findings separate from the individual reference descriptions: a reader should get the takeaway without reading every source.

## 2. Questions to answer

A research note is useful when it answers the questions a decision-maker will ask. Pick the ones that apply to the topic:

- What is this, and why does it matter to the project?
- What are the options or approaches, and how do they compare?
- What did prior work do, and how does this project relate (similar, complementary, superseded)?
- What was considered and rejected, and why?
- What does this imply for the design — what does it constrain or enable?

Not every note answers all of these; answer the ones relevant to the topic.

## 3. Common types

A research doc is usually one of these. The types are a guide to intent, not a fixed template — pick the one that fits, or combine several in one doc divided by topic.

- **Background** — get to know what this project is trying to do: the domain, motivations, and context a reader needs before the design makes sense.
- **Related work** — what has been done before: existing systems, papers, or products, and how this project relates (similar, complementary, superseded).
- **Brainstorm / trade-off analysis** — what can be done for our project: the options considered and their trade-offs. It explores possibilities; it does not record the decision eventually adopted — that goes in the design doc's decision record.
