# Root files

The recommended root files for a project following the blueprint. Each file has a single role; projects add or skip files based on their stack.

## 1. Files copied for every project

These are stack-agnostic and ship with every scaffold.

### 1.1 README.md

First file a new reader opens. Two-paragraph max: what the project is, how to start. Points at `docs/index.md` for everything else. Do not duplicate doc content here ŌĆö `README.md` is the entry pointer, not a second doc tree.

### 1.2 CLAUDE.md

Top-level instructions for AI coding agents working in the repo. Points at:

- `docs/index.md` for the documentation map.
- `docs/rules/code_general.md` (plus per-stack rule files) for code changes.
- `docs/rules/doc_markdown.md` for doc format; for content criteria on dynamic doc types, see the `aqr-content-criteria` skill.
- The issue workflow doc (if the project uses `aqr-issue-records`).

Also carries toolchain specifics ŌĆö language versions, where the version file lives, anything that surprises a fresh agent (system Node being the wrong version, pyenv env name, required system packages). These notes go here, not in `docs/`, because they affect agent behavior directly.

### 1.3 .editorconfig

Editor-agnostic style defaults. LF line endings, UTF-8, 4-space indent for Python, 2-space indent for JS-family and config files, tabs for Makefile. Editors that respect `.editorconfig` apply these automatically.

### 1.4 .gitignore

Common ignore patterns: local env files, editor state, caches, build output. The shipped template is a starting point ŌĆö projects uncomment `/data/` and `/logs/` lines if they carry data or log files in the working tree.

### 1.5 .gitattributes

Forces LF line endings on checkout regardless of contributor platform. Marks binary file types so they are never diffed as text. Add project-specific binary types as needed.

### 1.6 .markdownlint.json

Disables markdownlint rules incompatible with the project doc style: tight list-spacing (MD007), hard line-length limit (MD013), blank-line requirements around headings/lists (MD022, MD031, MD032), duplicate-heading within a document (MD024), etc.

## 2. Files added per stack

### 2.1 Node projects

- `.nvmrc` ŌĆö pins Node version. Read by `nvm` and similar version managers. Single line: the version number, no `v` prefix.
- `package.json` ŌĆö npm manifest. The `engines.node` field should match `.nvmrc`. The `packageManager` field is set when the project has chosen a specific manager (npm / pnpm / yarn); otherwise omit and let contributors use whatever is installed.

Lockfiles (`package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`) are generated by the package manager and committed. Do not hand-edit.

### 2.2 Python projects

- `.python-version` ŌĆö pins Python version. Read by `pyenv` and similar version managers. Single line: the version number.
- `pyproject.toml` ŌĆö Python project metadata, build config, and tool config (ruff, pyright, pytest). The recommended layout keeps tool config here rather than in separate files (`pytest.ini`, `.ruff.toml`, `pyrightconfig.json`) so there is one source of truth.

Requirements files (`requirements.txt`, `requirements-dev.txt`) are optional and project-specific. If the project uses `uv` or `poetry`, lock files (`uv.lock`, `poetry.lock`) are committed.

### 2.3 Other stacks

Add stack-specific root files following the same principle: one file per role, version pinning at the root, tool config centralized in one file when the tool supports it.

## 3. Files to skip

Do not create these unless the project explicitly needs them:

- `LICENSE` ŌĆö added per-project based on the chosen license; do not scaffold a default.
- `CHANGELOG.md` ŌĆö only if the project commits to maintaining one.
- `CONTRIBUTING.md` ŌĆö only if the project expects external contributors. Otherwise the rule docs under `docs/rules/` cover this.
- `.pre-commit-config.yaml` ŌĆö only if the project uses pre-commit. The blueprint assumes linting happens at edit and commit time via editor / agent discipline, not via pre-commit hooks.
